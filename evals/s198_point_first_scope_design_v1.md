# S198 — point-first, support-closed obligations before a scope-bound question

## Recommendation and measured target

S198 should replace the joint S197 author step with a two-stage author boundary:
first select and phrase two to four source-supported obligations with an explicit
generic facet; only after those obligations pass a cross-provider semantic screen may
a separate economic call write one Spanish field question whose scope is exactly that
accepted set. This remains upstream of the S193 planner.

The current comparable fact picture remains 157 facts: 143 OK, 12 synthesis-miss and
2 retrieval-miss (91.08% OK, seven facts short of 95%). Synthesis is still the dominant
residual bucket. S197 moved zero facts and did not measure the planner. Its author
transport passed 14/14 with zero invalid outputs, but Luna rejected 12/14 items:
8 question/point-set completeness failures, 5 unsupported or question-irrelevant
points and 6 wrong facets. The primary S198 endpoint is therefore not facts or planner
recall. It is qualification of the complete point-first package on a clean,
question-bounded diagnostic cohort, with zero failures in all three upstream classes.
S198 changes the author dependency, facet boundary contract and two semantic screens
together; it does not isolate their individual causal effects. A GO may authorize a
separate S199 planner preregistration; it does not itself move facts.

## Package hypothesis

S197 made one model choose a question and a maximum of four answer points in the same
response. A broad question could entail more obligations than the rectangular point
budget, while facet assignment and claim/support closure were checked only after the
whole cohort had been authored. The dominant failure is the direction of dependency,
not schema compilation: the question should be a rendering of an accepted obligation
set, not an upstream source of unbounded obligations.

S198 qualifies this complete upstream package:

1. select a small cohesive obligation set directly from source units;
2. screen each obligation for support, materiality and facet fit;
3. render a question from only that accepted set;
4. screen bidirectional scope equivalence between question and set.

The package does not qualify if a wholly fresh cohort produces any invalid,
unsupported, wrong-facet, incoherent, trivial or scope-mismatched item. Zero failures
is a deliberately conservative GO rule, not statistical falsification: a single Luna
disagreement is weak evidence about the mechanism and is reported by failure class,
not promoted to a general causal conclusion. No S197 item is repaired, retried,
replayed or used as an example.

## Stage A — static point plan, no question

The economic point author remains pinned Haiku 4.5. It reuses the exact S196 static
rectangular transport already compiled by Anthropic: eligibility, a required question
string and four always-present point slots with three always-present support strings.
For this stage the question field must be the empty string for both eligible and
ineligible items; a deterministic S198 normalizer enforces that rule. Thus the provider
schema remains source-agnostic and unchanged, while S198 changes only the task and
post-parse contract.

The prompt supplies the sealed source identity, all evidence units, allowed unit IDs
and a versioned facet catalogue. The labels are the S197 labels; the exhaustive
precedence below is a declared S198 contract change motivated only by S197's aggregate
wrong-facet bucket, never by item-level wording or repairs:

- `access_or_prerequisite`: permission, dependency or state required before the task;
- `target_or_configuration_field`: address, terminal, parameter or value to set;
- `input_trigger_or_observed_condition`: event or observed state that activates logic;
- `output_action_or_corrective_step`: commanded behavior or technician remediation;
- `option_mode_or_default`: selectable operating mode, alternative or default;
- `measurement_limit_or_timing`: measured value, tolerance, capacity, range or time;
- `safety_warning_exception_or_conflict`: hazard, prohibition, exception or conflict;
- `verification_commissioning_or_recovery`: test, confirmation, commissioning or restore step.

Each point must be one atomic claim. If it contains independently actionable clauses
with different best-fit facets, the author must split it or mark the item ineligible.
For a genuinely overlapping atomic claim, the first applicable class wins:

1. safety hazard, prohibition, exception or conflict;
2. verification test, confirmation, commissioning or restore;
3. corrective technician remediation;
4. prerequisite permission, dependency or prior state;
5. numeric measurement, tolerance, capacity, range or time;
6. configuration address, terminal, parameter or value to set;
7. option, operating mode or default;
8. trigger or observed condition.

These definitions and tie-breaks are generic and frozen before new source selection;
no manufacturer, product or S197 item wording enters them.

Eligibility has one operational definition for author and judge: the sealed evidence
units must support two to four atomic, semantically distinct and materially useful
obligations which together form one coherent, non-trivial and natural field-technician
question about the bound product without outside knowledge. Otherwise the item is
ineligible.

Deterministic validation requires the same membership, contiguity, cardinality,
inactive-slot, distinct-claim and receipt reconstruction invariants as S197. Eligible
items contain two to four points; each point cites one to three unique known units and
uses exactly one catalogue facet under the precedence above. Ineligible items contain
no point content.

## Stage A semantic gate — support and usefulness before rendering

Only a structurally valid point plan reaches Luna `gpt-5.6-luna`, reasoning `none`,
`store=False`. Luna sees the complete excerpt units and judges, separately:

- eligibility correctness;
- exact claim entailment by the cited units;
- cited-support relevance and sufficiency;
- best-fit facet under the frozen definitions;
- semantic distinctness of points;
- material usefulness for a field technician;
- whether the point set forms one coherent, non-trivial question-worthy task bundle.

Every false judgement requires an issue and every true/null judgement requires none.
Any invalid review or false judgement is an immediate `NO_GO_POINT_PLAN_SEMANTIC_GATE`.
Question writing is not called for a failed cohort. Luna is a screen, not a calibrated
gold authority; judge accuracy and human agreement remain `NOT_MEASURED`.

## Stage B — question rendered from accepted points only

The question writer is a second Haiku 4.5 call with a new minimal static schema:
`item_id` plus `question`, both strings, with no dynamic enum/const. It receives product
identity and only the accepted claims/facets — not the excerpt, unused units, S197
outputs or Luna reasoning. It must write one natural Spanish technician question for
which every accepted point is necessary and which requests no obligation outside that
set. Exact item identity, non-empty/length bounds and absence of evaluation/meta wording
are checked deterministically.

Because this is a new provider schema, a separate synthetic canary must compile and
round-trip it once before any fresh-document S198 packet is selected or any real-cohort
model is called. Canary success is only a transport GO and moves zero facts.

Each valid question receives a final Luna screen, bound to the accepted points and the
original excerpt. The strict output judges:

- Spanish language and field-technician naturalness;
- answerability from the excerpt;
- every accepted point is required by the question;
- no question-implied obligation is missing from the accepted set;
- the question does not widen product, condition, procedure or safety scope;
- the bundle remains coherent and non-trivial after rendering.

Any invalid output or false judgement is `NO_GO_QUESTION_SCOPE_GATE`. A GO requires
zero point-plan and zero question-scope failures; thresholds are not percentages.

## Fresh population and anti-overfit contract

After the synthetic canary and before any real-cohort model call, a new GET-only packet
must perform two identical complete `chunks_v2` scans and select exactly 14
documents/manufacturers, seven table and seven prose. It excludes every S194, S195 and
S197 document, source file,
manufacturer/product pair, protected target UUID and exact protected-target
content/extraction hash. Every protected target UUID must resolve. The evidence-unit
manifest is sealed before authoring. Semantic near-duplicate and OEM relabel overlap
remain `NOT_MEASURED`; the result must not claim otherwise.

The source receipt must report the eligible inventory remaining after all exclusions:
document count, source-file count, manufacturer/product-pair count and manufacturer
count, plus all selected identities and the corresponding post-selection reserve. No
claim of an infinite reserve or of 30 available manufacturers is permitted without
those physical counts.

The real cohort is selected only after the canary and after design, facet definitions,
schemas, gates, budgets and tests are versioned and reviewed. The cohort is executed
once. There is no
same-cohort retry, repair, prompt edit, threshold edit or rescore. Aggregated S197 counts
motivate the seam and the declared generic facet precedence; S197 questions, claims,
manufacturers and issue text are forbidden from prompts, fixtures and decision rules.

Population minima remain at least 12 eligible manufacturers/questions, at least five
table and five prose questions and at least 24 accepted points. Additionally GO requires
zero invalid author outputs, zero invalid reviewer outputs, zero unsupported claims,
zero wrong facets, zero non-material/incoherent bundles and zero question-scope
mismatches. These gates are frozen before source selection.

## Execution, receipts and states

Frontier models are used only for design and critical review. Execution is local,
deterministic where possible and otherwise freezes Anthropic SDK `0.97.0`, OpenAI SDK
`2.30.0`, point author and question writer model
`claude-haiku-4-5-20251001`, and both semantic-screen calls on `gpt-5.6-luna` with
reasoning `none` and `store=False`. Output-token caps are 1,200 for each point author,
300 for each question writer, 700 for each point screen and 500 for each question
screen.

Maximum real-cohort inference calls are 14 point authors + 14 point screens + 14
question writers + 14 question screens = 56, with `max_retries=0`. At most one
token-count preflight per inference call yields at most 56 preflights and 112 total
provider requests. The frozen accounting prices are Haiku $1 input/$5 output and Luna
$1 input/$6 output per million tokens; the auditable internal ceiling is $3, well below
the user's standing $250 limit. Clients are constructed only after an exclusive
workspace lock, immutable prepaid checkpoints and exact SDK/model verification.

Every response persists provider ID/model/status, raw output, normalized output,
usage, input/schema hashes and physical artifact hashes. Progress replacement is
same-directory atomic. A provider interruption before a known content failure is HOLD;
after any known deterministic or semantic failure it is NO-GO. A provider/schema 400
is a stage-specific rejection, not evidence for a broader compiler claim. A losing
lock process cannot seal over the winner. The lock is workspace-local, not distributed.

State machine:

- `GO_QUESTION_SCHEMA_CANARY_COMPILED`: synthetic schema transport only;
- `GO_POINT_FIRST_SCOPE_BOUND_COHORT_SEALED`: all fresh upstream gates pass;
- `NO_GO_POINT_PLAN_*`: structural or semantic point-plan failure;
- `NO_GO_QUESTION_*`: question transport or scope failure;
- `HOLD_EXTERNAL_DEPENDENCY_INCOMPLETE`: no known content failure and external interruption;
- unexpected post-lock failures seal HOLD/NO-GO with known-failure precedence.

Only the cohort GO may authorize a separate S199 planner contract with recall ≥90%,
precision ≥80% and complete questions ≥75%, plus exactness, deterministic checks, zero
covered-obligation regressions and zero new versioned conflicts. S198 does not execute
that planner, protected targets, retrieval, reranking or runtime integration.

## Alternatives considered

- **Prompt-tune the joint S197 call:** rejected because it reuses the wrong dependency
  and invites wording-level overfit to 14 observed failures.
- **Raise the four-point cap:** rejected because broad questions can always outrun a
  larger finite cap and the S193 planner target is bounded decomposition, not exhaustive
  document summarization.
- **Let Luna repair claims/facets/questions:** rejected because the validator would
  silently become the author and erase cross-provider screening independence.
- **Assign facets with keyword rules:** rejected because facet boundaries are semantic;
  deterministic keywords would encode brittle vocabulary and manufacturer bias.
- **Open S193/S199 planner on the 2 passing S197 items:** rejected because that is a
  post-selected denominator and cannot support generalization.
- **Use a frontier model for the cohort:** rejected because the mechanism, not maximum
  model spend, is the variable; frontier calls are reserved for design review.

## Known limits and explicit lanes

The design can screen for question-bounded completeness but cannot ensure it with one
uncalibrated Luna judgement; nor does it cover every technical opportunity in an
excerpt or representative real-user difficulty. If question rendering fails, the
sealed result must distinguish an apparent dependency/rendering failure from possible
context starvation caused by deliberately withholding the excerpt from the writer.
Document-wide facts,
multi-document contradictions, OEM relabels, country profiles, OCR/diagram-only loss,
seven-segment interpretation, translation coverage, Luna calibration and human
agreement remain `NOT_MEASURED`. A GO creates a diagnostic cohort, not gold or a
production KPI.

The current fact denominator remains hybrid with 77 legacy carries; S172 and S188 stay
integrated locally/default-off pending independent generalization. `chunks_v2` remains
the active baseline. `chunks_v3` remains the explicit
`FINAL_NO_GO_CHUNKS_V3_WHOLESALE` lane and is neither read as a candidate nor
rematerialized or patched. Railway is a demo and is not a PR or merge gate when CI is
green.
